# BothSafe Prisma — datasource and database roles

This directory holds the Prisma schema and (later) migrations for the BothSafe
deal flow. The datasource provider is **PostgreSQL 16**; the runtime topology
is documented in `/.kiro/steering/database.md`. The MySQL note in
`/AGENTS.md` is outdated — Postgres is the truth.

## Datasource

`schema.prisma` declares the `postgresql` provider; the connection URL is
supplied by `../prisma.config.ts` (Prisma 7 moved `url` out of the schema
into the CLI config file). At runtime the NestJS backend reads
`DATABASE_URL` directly when it constructs `PrismaClient`.

- Dev: `postgresql://bothsafe:bothsafe@localhost:55432/bothsafe?schema=public`
- Prod: `postgresql://bothsafe:${POSTGRES_PASSWORD}@postgres:5432/bothsafe?schema=public`

Both environments use the same database name (`bothsafe`) and schema
(`public`). Only the host, port, and password differ.

## Two database roles: `migrator` and `app`

The deal flow design (`.kiro/specs/bothsafe-deal-flow/design.md`,
section "Append-only enforcement (R14.2, R20.5)") requires that the
`wallet_ledger_entry` and `audit_log_entry` tables be **append-only at the
database level** — not merely in application code. To make that possible
without breaking schema migrations, we run Prisma against the database with
two distinct roles:

| Role        | Purpose                                  | Privileges                                                      |
| ----------- | ---------------------------------------- | --------------------------------------------------------------- |
| `migrator`  | Schema changes (DDL): `prisma migrate …` | Full owner privileges on the `bothsafe` database and `public` schema. Used only by migration tooling. |
| `app`       | Runtime queries (DML): NestJS backend    | `SELECT`, `INSERT`, `UPDATE`, `DELETE` on most tables. `UPDATE`, `DELETE`, `TRUNCATE` are **revoked** on `wallet_ledger_entry` and `audit_log_entry`. A `BEFORE UPDATE OR DELETE OR TRUNCATE` trigger calls `reject_mutation()` as a defence-in-depth fallback. |

The privilege revocation and triggers are applied by a migration owned by
`migrator` (see task 2.10 in
`.kiro/specs/bothsafe-deal-flow/tasks.md`). The application connection
string used by the NestJS process must point at the `app` role, so any code
path that tries to mutate an append-only row fails loudly at the database
boundary.

### Role wiring

- `DATABASE_URL` — used by the running backend; **must** authenticate as the
  `app` role.
- `MIGRATE_DATABASE_URL` (added later, when migrations are first run) — used
  by the Prisma CLI; authenticates as the `migrator` role. When unset in dev,
  it is acceptable to run migrations as the same user the dev compose
  bootstraps (`bothsafe`), provided the privilege-revocation migration in
  task 2.10 runs against a real `app` role afterwards.

In dev, `docker-compose.yml` provisions a single `bothsafe` superuser; the
`app`/`migrator` split is created by a post-migration SQL script
(`./sql/append_only_enforcement.sql`, applied by task 2.11) that:

1. Creates the `migrator` and `app` roles if they don't exist.
2. Grants `migrator` ownership of the `bothsafe` database / `public` schema.
3. Grants `app` `CONNECT` on the database and `USAGE` on the schema.
4. Sets the default privileges so future tables grant
   `SELECT, INSERT, UPDATE, DELETE` to `app`.
5. Revokes `UPDATE, DELETE, TRUNCATE` from `app` on the append-only tables
   and installs the `reject_mutation()` triggers.

See [`./sql/README.md`](./sql/README.md) for the recommended `psql`
invocation, idempotency notes, and verification steps.

## Migration commands (reminder)

- Dev: `npx prisma migrate dev` — generates SQL, applies it, may reset.
- Prod: `npx prisma migrate deploy` — applies existing migrations only,
  never generates, never resets.

Never run `prisma migrate reset`, `db push --force-reset`, or
`docker compose down -v` against production without explicit user
confirmation in the same turn (per
`/.kiro/steering/database.md`).


## Dev defaults for `append_only_enforcement.sql`

The post-migration script `./sql/append_only_enforcement.sql` reads two
passwords from the environment when it `CREATE ROLE`s the `migrator` and
`app` roles. For local development, the placeholder values used by the
project are:

- `MIGRATOR_PASSWORD=dev-migrator-pwd`
- `APP_PASSWORD=dev-app-pwd`

These are dev-only — they are seeded into `backend/.env.example` and
`backend/.env` purely to document them, never used to connect from the
running backend. Production deployments MUST override both env vars to
real secrets sourced from the deployment secret manager before running
the script.

## Initial migration (task 2.11)

The first migration was generated by

```bash
cd backend
npx prisma migrate dev --name init_deal_flow
```

which produced `prisma/migrations/<timestamp>_init_deal_flow/`. After
the migration applies, `./sql/append_only_enforcement.sql` is run via
`psql` (see `./sql/README.md`), and the seed script `./seed.ts` is run
via `npx prisma db seed` to provision a single platform user plus an
escrow `Wallet` (+ `WalletRole='escrow'`) per supported `Currency`.
The seed is idempotent — every step uses `upsert` against a stable
unique constraint.
